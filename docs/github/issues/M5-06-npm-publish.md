## Summary
Publish `@genie/server` and `@genie/viewer` to
npm on every release-please-generated release. Enable `--provenance`
(npm supply-chain attestation). Reserve scope `@genie`.

## Context
- INDEX.md: distribution = npm + `.mcpb` + Docker.
- Research report §7 step 1 + §5 prior art (Framelink uses `npx
  figma-developer-mcp`).
- M0-04 fallback (recorded in `CLAUDE.md` hard rule 7): the bare `genie` name
  is already registered on npm by an unrelated package, so the server
  publishes as **`@genie/server`** instead of bare `genie` — both packages
  are scoped under `@genie`.

## Acceptance Criteria
- [x] AC1 — the release workflow's publish jobs fire deterministically off
      every release-please-created release. **Reconciled wording:**
      `release-please-config.json` sets `include-component-in-tag: true`, so
      tags are `packages/server-vX.Y.Z` / `packages/viewer-vX.Y.Z`, not a
      bare `v*`. `.github/workflows/release.yml` gates each publish job on
      the `release-please` job's own per-package `release_created` output
      (the mechanism release-please's docs recommend) rather than a separate
      `on: push: tags: v*` trigger, which would race the tag-creation step
      itself. This satisfies AC1's intent (publish exactly when
      release-please cuts a release) without a literal `v*` filter.
- [x] AC2 — Workflow publishes both packages with `--provenance` flag set.
- [x] AC3 — `NODE_AUTH_TOKEN` sourced from GitHub Actions `NPM_TOKEN`
      secret with publish scope.
- [x] AC4 — Server publishes as `@genie/server` (scoped, public) — see the
      M0-04 fallback under Context.
- [x] AC5 — Viewer publishes as `@genie/viewer` (scoped, public).
- [x] AC6 — Publish step is gated on M0-04 CI green: a `verify-ci-green` job
      polls the GitHub Actions *workflow run* API for the `CI` workflow
      (`ci.yml`) filtered to the exact commit SHA being released, and reads
      that whole run's top-level `status`/`conclusion` (the same signal
      branch protection uses) rather than sampling individual name-matched
      check-runs. It fails closed if no run is found, the run hasn't
      completed within a 20-minute poll budget (sized above `ci.yml`'s own
      longest job `timeout-minutes: 20`), or the run's conclusion isn't
      `success`. Both publish jobs `need` it in addition to `release-please`.
- [x] AC7 — Release notes auto-generated by `release-please`.
